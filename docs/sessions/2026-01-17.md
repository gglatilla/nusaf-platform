## Decisions Made

### Decision 1: Replace Slash Commands with Specific Commands
| Aspect | Detail |
|--------|--------|
| **What** | Removed `.claude/commands/` system, replaced with specific commands like `nusaf start` |
| **Why** | Slash commands only worked when Claude was started from inside the project directory. User works from parent `C:\Users\Guido\Projects` folder. |
| **Alternatives Rejected** | 1) Keep slash commands - rejected due to working directory requirement. 2) Natural language phrases - rejected in favor of specific commands for less ambiguity. |
| **Consequences** | User uses specific commands like `nusaf start`, `nusaf save`. Works from any directory. |

### Decision 2: Specific Commands over Natural Language (v2.1)
| Aspect | Detail |
|--------|--------|
| **What** | Changed from "let's work on nusaf-platform" to `nusaf start` |
| **Why** | Specific commands are less ambiguous, easier to remember, and trigger more reliably. |
| **Alternatives Rejected** | Natural language ("let's work on...", "save progress") - more flexible but ambiguous. |
| **Consequences** | Slightly less natural but more reliable and consistent. |

### Decision 3: Mandatory Startup Checklist
| Aspect | Detail |
|--------|--------|
| **What** | CLAUDE.md contains a mandatory checklist Claude must complete before starting any work |
| **Why** | Without explicit requirements, Claude might skip reading context files, leading to lost context between sessions. |
| **Consequences** | Every session starts with full context. Prevents redundant work. |

### Decision 4: Separate CURRENT-WORK.md
| Aspect | Detail |
|--------|--------|
| **What** | Created dedicated file for tracking exact work state |
| **Why** | Session logs are historical. CURRENT-WORK.md is live state - different purposes. |
| **Consequences** | Two files to maintain but clearer separation. |

### Decision 5: CODEBASE-OVERVIEW.md
| Aspect | Detail |
|--------|--------|
| **What** | Created dedicated file for tracking what's been built |
| **Why** | As project grows, Claude needs quick way to know what exists without reading all code. |
| **Consequences** | Must be kept updated as features are added. |

### Decision 6: Test-As-You-Go Development Workflow (v2.2)
| Aspect | Detail |
|--------|--------|
| **What** | Run tests after completing each logical unit of work, not just at commit time |
| **Why** | Testing after every line is too slow; testing only at commit time lets bugs compound. Testing after each logical unit catches bugs early while maintaining efficiency. |
| **Alternatives Rejected** | 1) Test after every line change - too slow, interrupts flow. 2) Test only at commit - bugs compound, harder to debug. |
| **Consequences** | Claude must run tests after each todo item/function/feature. If tests fail, must fix before moving on. |

### Decision 7: Code Simplicity Guidelines (v2.3)
| Aspect | Detail |
|--------|--------|
| **What** | 8 balanced guidelines for keeping code simple without cutting corners |
| **Why** | Need sensible defaults that work across most scenarios. Guidelines can be adjusted as project needs become clearer. |
| **Key Principle** | Simplicity ≠ cutting corners. Security, validation, and compliance are requirements, not extras. |
| **Consequences** | Claude follows these guidelines when writing code. User pushed for honest assessment rather than confident assertions. |
